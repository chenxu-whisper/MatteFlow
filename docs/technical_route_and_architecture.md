# MatteFlow 技术线路与技术架构说明

## 1. 文档目的

本文档用于说明 `MatteFlow` 项目的技术目标、技术线路、系统架构、核心处理流程与部署运行方式，帮助研发、测试和后续维护人员快速理解项目的实现思路与模块边界。

## 2. 项目定位

`MatteFlow` 是一套面向视频、图片和序列帧的透明化处理工具，目标是将绿幕素材、黑底特效素材以及部分可判别背景素材转换为可直接用于后续合成的透明结果。项目同时提供命令行入口与 Web GUI，覆盖批处理和可视化交互两类使用场景。

从产品目标看，项目重点解决以下问题：

- 绿幕场景下的人物、动物、发丝、毛发、羽毛等细节保留
- 黑底特效场景下的粒子、火焰、烟雾、辉光、能量光等半透明内容保留
- 压缩视频中的边缘污染、色块与溢色处理
- 视频或序列帧处理时的 Alpha 连续性与时序稳定性

## 3. 技术目标

项目当前的技术目标可概括为以下四点：

1. 支持视频、图片、序列帧三类输入，并统一进入同一条处理流水线
2. 支持自动识别背景模式，并在绿幕与黑底两类场景中走不同抠图策略
3. 支持传统算法与多种 AI 模型协同工作，在速度、质量和模型可用性之间做权衡
4. 支持预览、导出、队列化处理与本地部署，满足生产使用与桌面场景

## 4. 技术栈概览

### 4.1 基础语言与工程管理

- 语言：Python 3.10+
- 打包方式：`hatchling`
- 项目结构：标准 `src` 布局
- 依赖管理：同时支持 `pip` 与 `uv`

### 4.2 核心依赖

- 数值与图像处理：`numpy`、`opencv-python`、`pillow`
- 视频与媒体处理：`av`、`imageio`、`imageio-ffmpeg`、`pims`
- 深度学习框架：`torch`、`torchvision`
- 模型生态：`transformers`、`accelerate`、`diffusers`、`peft`、`timm`
- 配置与模型运行支撑：`hydra-core`、`omegaconf`、`huggingface-hub`
- 轻量推理与抠图：`onnxruntime`、`rembg`
- 交互界面：`gradio`
- GPU 可选依赖：`cupy-cuda12x`、`pynvml`、`triton`

### 4.3 技术风格

项目采用“单体 Python 应用 + 本地模型推理 + 分层模块化组织”的方式实现，不依赖独立前端工程，也不是云原生微服务架构。它更偏向本地桌面工具和单机生产工具的实现形态。

## 5. 技术线路

### 5.1 总体路线

MatteFlow 的技术线路不是单一抠图模型直出，而是围绕“可落地生产质量”构建的一条混合链路：

1. 先统一完成输入识别与帧解码
2. 再进行背景模式分析，决定后续抠图路径
3. 在抠图阶段使用传统算法与 AI 模型做可插拔组合
4. 在输出前补充边缘细化、去噪点、时序稳定、颜色去污染等质量增强环节
5. 最终统一编码为 RGBA、Alpha 相关结果和预览产物

这条路线的核心思想是：

- 用规则方法保证基础可用性和兜底能力
- 用 AI 模型提升复杂边缘和复杂透明区域的表现
- 用后处理链路修复模型输出在生产场景中的常见缺陷
- 用质量模式区分预览效率与最终导出质量

### 5.2 背景模式驱动路线

系统把输入素材按背景特征分成三类：

- `auto`：由背景分析模块自动判断
- `green_screen`：绿幕或绿底渲染素材
- `black_background`：黑底粒子、火焰、烟雾、光效素材

这意味着项目不是做“任意自然背景抠图”的通用方案，而是针对绿幕和黑底这两类生产场景做专项优化。该策略可以减少模型和算法范围，提高结果可控性。

### 5.3 混合算法路线

项目并未绑定单个模型，而是采用“传统算法兜底 + 多 AI 模型可选 + 自动回退”的混合路线。

当前接入或封装的能力包括：

- GVM
- MatAnyone2
- CorridorKey
- BiRefNet
- RVM
- SAM2
- rembg
- 传统绿幕抠图
- 传统黑底透明化

在 `ai_model=auto` 时，系统会按可用性和优先级尝试加载模型；当模型不可用时，仍可回退到传统算法。这样做兼顾了模型效果、环境兼容性和工具可运行性。

### 5.4 质量分层路线

项目通过 `fast`、`standard`、`high` 三档质量模式来控制性能与质量：

- `fast`：优先用于预览，跳过较重的后处理阶段
- `standard`：面向常规生产，启用主要细化与去噪能力
- `high`：面向高质量离线导出，强调边缘细节与稳定性

质量模式直接影响以下阶段是否启用或以何种强度运行：

- 边缘细化
- 去噪点
- 时序稳定
- 部分模型和分辨率相关参数

### 5.5 质量选择系统

MatteFlow 支持可选的质量选择系统。开启 `quality_selection_enable=True` 或 CLI 参数 `--quality-selection` 后，系统会生成多个候选 matte，按区域评估候选质量，并将候选决策摘要写入 `processing_report.json` 的 `quality_selection` 字段。默认配置保持关闭，以保证现有 CLI 和 GUI 行为稳定。

当前质量选择系统包含以下能力：

- 候选协议：所有候选输出统一为 `float32` alpha，范围裁剪到 `[0.0, 1.0]`，并校验尺寸。
- 候选来源：支持传统候选；MatAnyone2、BiRefNet、SAM2-guided 已有 wrapper，模型或 guidance 不可用时会结构化记录 `skipped_candidates`。
- BiRefNet 候选默认不会自动加载模型。需要真实 BiRefNet candidate 时，必须同时启用质量选择并显式打开 `quality_birefnet_auto_load` 或 CLI 参数 `--quality-birefnet-auto-load`。加载或推理失败会记录为 `skipped_candidates`，不影响 traditional fallback。
- 质量评估：基于 `RegionOwnership` 对主体、发丝边缘、发光道具、透明光效、背景残留和不确定边缘分别评分。
- 区域选择：`QualitySelector` 按区域选择得分最高的候选，并记录 `selected_model_counts` 和逐区域 `decisions`。
- 可观测性：开启 debug 时会输出 `debug/quality_selection/quality_selection_contact_sheet.png` 和质量选择 JSON 摘要。
- 回归能力：质量回归评估器会读取 `quality_selection.available` 与 `quality_selection.candidate_count`，防止开启质量选择但没有有效候选的退化。

### 5.6 本地化交付路线

项目采用本地推理与本地交互路线，关键特征包括：

- 模型权重本地下载并存放
- FFmpeg/FFprobe 本地发现或配置
- Web GUI 基于 Gradio 本地启动
- GPU 任务以单机串行队列执行

这一方案降低了服务端部署复杂度，更适合内容生产、特效制作和工具型交付。

## 6. 技术架构

### 6.1 分层架构

从代码结构上看，项目可以抽象为六层架构：

1. 接口层
2. 服务层
3. 调度层
4. 核心流水线层
5. 算法与增强层
6. 基础设施层

### 6.2 分层说明

#### 接口层

接口层负责承接用户输入与运行方式，包含两类入口：

- CLI：适合批处理、自动化脚本和研发调试
- Web GUI：适合上传素材、参数调节、预览和导出

接口层不直接持有复杂算法逻辑，而是把请求转交给服务层和流水线层。

#### 服务层

服务层以 `MatteFlowService` 为核心，对外提供稳定的任务处理接口，负责：

- 将外部输入整理为 `ProcessJobParams`
- 构造 `MattingConfig`
- 调用主流水线执行处理
- 将原始结果转换为统一的结构化结果
- 把运行时异常包装为统一错误语义

该层的价值是把 UI 和 CLI 与底层处理逻辑解耦，便于后续扩展更多入口。

#### 调度层

调度层以 `GPUJobQueue` 和 `JobWorker` 为核心，用于管理 GPU 重任务的排队与串行消费。其主要作用如下：

- 管理任务提交、去重、取消、历史记录
- 保证同一时刻只执行一个 GPU 重任务
- 为 GUI 提供稳定的任务状态和进度更新能力
- 区分正式处理、Alpha 生成、预览重处理等任务类型

该设计适合本地单机资源模型，避免多个推理任务同时争抢显存。

#### 核心流水线层

核心流水线层以 `MattingPipeline` 为主，是项目的业务主干。流水线按阶段组织处理过程，负责协调输入、分析、抠图、增强与输出，统一处理视频、图片和序列帧。

#### 算法与增强层

算法与增强层由多个子模块组成：

- `analysis/`：背景分析与自动模式判断
- `matte/`：传统抠图、AI 模型封装和融合调度
- `refine/`：边缘细化、去溢色、去噪点
- `temporal/`：视频时序稳定

这一层是项目质量能力的核心，也是技术积累最集中的部分。

#### 基础设施层

基础设施层负责支撑运行环境和输入输出能力，主要包括：

- `input/`：输入类型识别、视频/图片/序列帧解码
- `output/`：RGBA、遮罩和预览结果编码
- `utils/`：模型路径、模型检查、下载、兼容性和输出路径工具
- `ffmpeg_env.py`：FFmpeg 环境发现与配置
- `diagnostics.py`、`errors.py`：诊断与错误建模

## 7. 核心处理流程

### 7.1 主流程说明

主流程围绕 `MattingPipeline.process()` 组织，整体顺序如下：

1. 识别输入类型并完成解码
2. 在自动模式下分析背景类型
3. 生成 Alpha Matte
4. 按质量模式执行边缘细化
5. 按质量模式执行去噪点
6. 对视频或序列帧执行时序稳定
7. 基于稳定后的 Alpha 执行颜色去污染
8. 编码输出透明结果与相关产物

### 7.2 各阶段职责

#### 输入解码

输入解码层将文件或目录转换为统一的帧序列和元信息，屏蔽视频、单图、序列帧之间的差异，保证后续阶段只处理统一的数据结构。

#### 背景分析

当背景模式为 `auto` 时，背景分析器会基于采样帧特征判断素材更接近绿幕还是黑底，并将结果作为后续算法选择依据。

#### Matte 生成

Matte 生成阶段由 `HybridMatte` 负责，是整个系统的算法调度中心。该模块会根据配置决定：

- 是否启用 AI
- 使用指定模型还是自动选型
- 当前素材更适合走哪条抠图路径
- AI 模型不可用时如何回退

#### 边缘细化

边缘细化主要用于改善人物边界、发丝、羽毛、半透明轮廓等区域的 Alpha 质量，减少边缘发硬和断裂问题。

#### 去噪点

去噪点阶段重点清理抠图后产生的小块噪声和零散污染区域，提升最终遮罩可用性，降低后续合成中的脏点问题。

#### 时序稳定

时序稳定只在多帧场景中启用，目标是减轻连续帧间 Alpha 的闪烁、跳变和抖动，使视频和序列帧合成结果更稳定。

#### 颜色去污染

颜色去污染在 Alpha 稳定后执行，用于解决绿边、黑边、溢色和边缘发灰问题，使最终前景在不同背景上合成时更自然。

#### 输出编码

输出编码层负责把处理后的帧和 Alpha 编码为透明结果、遮罩、调试图以及 GUI 可消费的预览资源。

## 8. 核心模块说明

### 8.1 配置中心

`MattingConfig` 是系统统一配置模型，集中描述：

- 背景模式
- 质量模式
- AI 模型策略
- 绿幕参数
- 黑底参数
- 边缘与半透明保护参数
- 输出参数
- 性能参数

这使得 CLI、GUI、服务层和流水线层可以共享同一套配置语义。

### 8.2 AI 模型调度

`HybridMatte` 通过“显式指定 + 自动选择 + 兜底回退”的方式管理模型能力，其架构价值在于：

- 降低单模型绑定风险
- 让工程对模型缺失更有韧性
- 方便后续继续接入新模型
- 将算法选择逻辑集中收口，避免散落到各入口

### 8.3 服务与队列协作

CLI 与 GUI 共用同一套核心处理能力，但调用方式不同：

- CLI 直接调用服务层和流水线，适合一次性处理
- GUI 通过服务层结合队列执行，适合交互式预览和导出

这种设计让“处理能力”和“交互能力”分离，增强了架构复用性。

## 9. 目录与模块映射

项目目录可按职责理解为：

- `src/matteflow/analysis/`：背景检测与自动模式分析
- `src/matteflow/input/`：输入识别与解码
- `src/matteflow/matte/`：传统算法、AI 模型封装、融合与调度
- `src/matteflow/refine/`：边缘与颜色后处理
- `src/matteflow/temporal/`：时序稳定
- `src/matteflow/output/`：结果导出与编码
- `src/matteflow/vendor/`：内置第三方运行时代码
- `src/matteflow/service.py`：服务层统一入口
- `src/matteflow/job_queue.py`：GPU 任务队列
- `src/matteflow/job_worker.py`：队列消费执行器
- `scripts/web_gui.py`：Web GUI 启动与交互逻辑
- `scripts/download_models.py`：模型下载
- `scripts/configure_ffmpeg.ps1`：FFmpeg 环境配置

## 10. 部署与运行架构

### 10.1 运行形态

当前项目主要有两种运行形态：

- 命令行形态：通过 `python -m matteflow` 直接处理素材
- 本地 Web 工具形态：通过 `python scripts/web_gui.py` 启动 Gradio 服务

### 10.2 依赖条件

运行项目需要重点关注以下依赖条件：

- Python 环境与依赖版本
- PyTorch 与 CUDA 匹配关系
- 模型权重是否已下载
- FFmpeg/FFprobe 是否可发现
- 本地 GPU 显存是否足够

### 10.3 架构边界

当前部署方式是“单机本地计算 + 本地界面访问”，因此具备以下边界特征：

- 不包含远程任务编排系统
- 不包含分布式推理与水平扩展设计
- 不包含独立对象存储和数据库依赖
- 不包含前后端分离部署链路

这意味着项目当前架构更强调可运行、可维护和本地生产效率，而不是大规模服务化扩展。

## 11. 架构优势与风险

### 11.1 当前优势

- 分层清晰，接口层、服务层、流水线层边界明确
- 混合抠图方案兼顾效果、性能和环境适配
- 质量模式分级合理，适合预览与导出两类场景
- 本地化部署简单，便于工具交付与生产使用
- 通过队列串行化 GPU 任务，降低资源冲突概率

### 11.2 主要风险

- 模型依赖较多，环境配置复杂度较高
- 多模型并存带来维护成本和兼容性压力
- 本地单机串行队列在高并发场景下扩展性有限
- FFmpeg、CUDA、模型路径等外部依赖是常见故障点
- Web GUI 当前仍是本地工具架构，远程协作能力有限

## 12. 后续演进建议

从当前架构看，后续可沿以下方向演进：

1. 继续统一模型接入协议，降低新增模型成本
2. 抽象更明确的算法能力层，进一步弱化具体模型实现耦合
3. 为 GUI 与服务层增加更强的任务持久化与可观测性
4. 完善模型资产管理、依赖自检与环境诊断能力
5. 在保持本地工具形态的基础上，为未来服务化部署预留扩展点
